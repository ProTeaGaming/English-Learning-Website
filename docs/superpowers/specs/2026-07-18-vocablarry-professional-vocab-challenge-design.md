# VocabLarry Professional Environment — Vocab Quiz (Challenge mode) (design)

## Context

This is sub-project 2d of the VocabLarry Professional Environment rebuild
(see `docs/superpowers/specs/2026-07-17-vocablarry-professional-foundation-design.md`,
`docs/superpowers/specs/2026-07-17-vocablarry-professional-vocab-browse-word-design.md`,
`docs/superpowers/specs/2026-07-18-vocablarry-professional-vocab-quiz-design.md`,
and `docs/superpowers/specs/2026-07-18-vocablarry-professional-vocab-gap-design.md`
for parent context and locked-in architecture decisions). Foundation, Vocab
Browse + Word, Vocab Quiz (Quiz mode), and Vocab Quiz (Gap mode) are all
merged to `main`.

The production `VocabLarry/` SPA's Test page has 3 top-level modes — Quiz,
Gap, and **Challenge** (a random mix of everything). This spec covers
Challenge mode, the last of the 3 families and the final sub-project needed
to reach full parity with production's quiz engine (Gap mode's own
`Mixed Review` and Quiz mode's own `Mixed Review` are each already done —
Challenge is a third, separate kind of "mixed" that spans *across* both
families rather than staying within one).

In production, Challenge mode has **no sub-mode picker at all** — selecting
it just hides both the Quiz and Gap mode-picker areas, and every question
is generated by `buildHybridQuestion(word)`, which picks one question type
uniformly at random per word from `{gap, definition, word, synonym?,
antonym?}` (synonym/antonym only included when the word has any). This is
architecturally the simplest of the 3 sub-projects: it introduces exactly
one new function, reuses every question-generation and rendering code path
Quiz and Gap modes already built, and needs no new backend (same as both
prior quiz sub-projects).

One production quirk is deliberately **not** replicated (see Decisions):
production's `buildHybridQuestion` includes `"gap"` in the candidate list
unconditionally, calling `buildGapQuestion(word)` with no `gapMode` argument
(always defaulting to the plain "context" sub-type) even when the target
word has no `gap` sentence at all — production DB data shows only ~20% of
words (1,000 of 5,000) have one. When a Challenge question lands on a
gap-less word, production renders a blank-text question. The rebuild
guards against this the same way it already guards synonym/antonym
inclusion (only offer a candidate when the word can actually support it).

## Decisions

- **No new backend, no new URLs.** Challenge mode is the third value the
  existing family toggle can produce (Quiz mode's toggle already ships;
  Gap mode's toggle already ships) — same `/vocab/quiz/` and
  `/vocab/quiz/play/` routes/views, same `/api/words/`/`/api/categories/`
  data source as both prior quiz sub-projects.
- **Setup page: a third family radio, no new sub-mode picker.** The
  existing Quiz/Gap radio toggle gains a `value="challenge"` option.
  Selecting it hides/disables both the Quiz mode `<select>` and the Gap
  sub-mode `<select>` — matching production's `renderTestModes`, which
  shows nothing in their place. Since no visible control exists to carry a
  `mode` value when Challenge is active, a third form element is added: a
  hidden `<input type="hidden" name="mode" value="challenge">`, toggled
  enabled/disabled by the same script that already manages the two
  `<select>`s, so exactly one of the three `mode`-carrying elements is
  ever enabled and the form still submits a single `mode` value.
- **`mode` stays a single query-string param.** Challenge's value is the
  flat string `challenge` (no sub-mode suffix, since it has no sub-modes) —
  consistent with Quiz's flat values (`definition`, `mixed`, etc.) and
  Gap's `gap-`-prefixed values.
- **No change to `buildPool()`.** `"challenge"` doesn't match the existing
  `synonym`/`antonym`/`gap-`-prefix branches, so it already falls through
  to the plain category/CEFR-filtered pool — the same pool Quiz mode uses.
  This matches production's own `testPool()`, which likewise has no
  Challenge-specific branch.
- **New `buildHybridQuestion(word)`**, called once per target word inside
  `generateQuestions()`:
  - Candidate types always include `definition` and `word`; `synonym` and
    `antonym` are added only if the word has at least one (matching the
    exact guard Quiz mode's own `randomMixedMode` already uses); `gap` is
    added only if the word's `gap` field is non-empty and contains `___`
    (the one deliberate deviation from production, explained above — this
    guarantees Challenge never renders a blank-text gap question).
  - One candidate is picked uniformly at random from whatever ended up in
    the list (always at least 2 — `definition` and `word` are
    unconditional, so the list is never empty).
  - If the pick is `gap`, a gap sub-mode is picked uniformly at random from
    all 4 concrete sub-modes (`gap-context`/`gap-nuance`/`gap-collocation`/
    `gap-connotation`) and `buildGapQuestion(word, subMode)` is called —
    **full variety**, not production's context-only default, per the
    resolved design question (the rebuild's `buildGapQuestion` already
    handles this cleanly via its own `gap-mixed` resolution, so reusing it
    here is a trivial, low-risk improvement rather than a new algorithm).
  - Otherwise `buildQuestion(word, pick)` is called — the exact same
    function Quiz mode already uses, unmodified.
  - `buildHybridQuestion` returns whatever `buildGapQuestion`/
    `buildQuestion` returns, with no wrapping — a gap pick already carries
    `type: 'gap'` from Gap mode's existing return shape, and a quiz-style
    pick carries no `type` field, exactly like a normal Quiz-mode question.
- **No changes needed to `renderQuestion()` or `handleAnswer()`.** Because
  `buildHybridQuestion` returns objects shaped identically to what
  `buildGapQuestion`/`buildQuestion` already produce, the existing
  `q.type === 'gap'` feedback branch (added in Gap mode) and the standard
  rendering path handle Challenge questions automatically — this sub-project
  adds zero new rendering/feedback code.
- **`generateQuestions()` gains one new branch**, checked before the
  existing `gap-` prefix check: `if (mode === "challenge") return
  buildHybridQuestion(word);`.
- **No score/progress persistence** — same as Quiz and Gap modes.
- **No dialect substitution, no Vietnamese content translation** — same
  exclusions as prior phases.

## Architecture

```
templates/
  vocab/
    quiz_setup.html   (add Challenge radio + hidden mode input)
static/
  js/vocab-quiz.js     (add buildHybridQuestion + one dispatch branch)
```

No changes to `config/urls.py`, `config/views_vocab.py`, `quiz_play.html`,
or `static/css/vocab.css` — Challenge introduces no new visible UI element
beyond the one additional radio label (which reuses the existing
`.vocab-quiz-family-toggle label` styling already shipped in Gap mode) and
no new question-card markup.

## Components

- **`quiz_setup.html`** — the family toggle's `<div class="vocab-quiz-
  family-toggle">` gains a third `<label><input type="radio" name="family"
  value="challenge" id="familyChallenge"> Challenge</label>`. Inside the
  `<form class="vocab-quiz-fields">`, a new `<input type="hidden"
  name="mode" value="challenge" id="challengeModeInput" disabled>` sits
  alongside the two existing mode `<select>`s. The existing inline
  `applyFamily(family)` script gains one more branch: when `family ===
  "challenge"`, both `quizField`/`gapField` are hidden and both
  `quizSelect`/`gapSelect` are disabled (same as today when the *other*
  family is picked), and `challengeModeInput.disabled` is set to `false`
  only in this case.
- **`static/js/vocab-quiz.js`** — extended, not restructured:
  - New `buildHybridQuestion(word)` per the Decisions section above.
  - `generateQuestions()` gets the one new `mode === "challenge"` branch.
  - `buildPool()`, `renderQuestion()`, `handleAnswer()`, `renderResults()`
    are all unchanged.

## Data flow

`GET /vocab/quiz/` → user picks "Challenge" family (no further
configuration possible), plus category/CEFR/count as usual → form submits
`mode=challenge&category=...&cefr=...&count=...` → `vocab_quiz_play`
renders the same shell as Quiz/Gap modes → `vocab-quiz.js` fetches
`/api/words/` once → `buildPool()` returns the plain category/CEFR-filtered
pool (no sub-type filter) → `generateQuestions()` calls
`buildHybridQuestion` per target word, each independently landing on a
different question type → renders/answers/results exactly as the existing
flow does, with each question's type varying question-to-question. "Try
Again" and "Change Settings" behave identically to Quiz and Gap modes.

## Error handling

Same as Quiz and Gap modes: an empty category/CEFR-filtered pool shows the
existing "No words available for this combination — try different
settings" message. No new error case is introduced — a word landing in
the pool always has at least `definition`/`word` available as a fallback
question type, so no individual question can fail to generate once a word
is selected.

## Testing

Same split as Quiz and Gap modes. Python/pytest coverage: new tests assert
the Challenge radio (`value="challenge"`) and the hidden mode input appear
in the setup page's rendered HTML — extending `tests/test_vocab_pages.py`
in place. No server-side equivalent of `buildHybridQuestion` to unit test.

`buildHybridQuestion`'s logic is verified via `node --check` for syntax
plus a direct read-through of the ported candidate-selection/fallback
logic against this spec's exact rules — the same rigor Gap mode's review
applied, since (as with Gap mode) there is no automated test catching a
logic error here. Manual browser testing covers: running a Challenge quiz
and observing multiple distinct question types across several questions
(including at least one gap-type question, given the ~20% gap-eligibility
rate in the dataset), correct scoring and feedback for both a quiz-style
and a gap-style question within the same run, a category/CEFR combination
whose words mostly lack synonyms/antonyms/gap sentences still producing
valid definition/word questions, and Try Again / Change Settings both
working from a Challenge-mode run.

## Explicitly out of scope for this phase

Free-typed (non-multiple-choice) answer input, hand-picking individual
words as a quiz source, learned-state filtering, section grouping,
Vietnamese content translation, dialect-based word substitution, and any
score/progress persistence. This is the last of the 3 quiz-family
sub-projects (Quiz, Gap, Challenge) — remaining future work on the Vocab
section is Grammar and Dashboard, which are separate, not-yet-started
sub-projects.
