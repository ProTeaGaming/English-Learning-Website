# Vocabulary US/UK English Variant — Design

**Date:** 2026-07-16
**Product:** Django only (`VocabLarry/vocablarry.html`)
**Status:** Approved by user (conversation, 2026-07-16)

## Purpose

Split "English" into two selectable variants — English (US), the existing
default, and English (UK) — so learners can study the vocabulary content
in British spelling where it genuinely differs (colour, flavour,
sceptical, kerb, endeavour, etc.). This is a content/spelling toggle for
the vocabulary being *taught*, not a UI translation language — it is a
sibling of the existing English/Vietnamese switcher, not a replacement
for it.

This is **Phase 1** of a two-phase project. Phase 1 covers the Vocabulary
word dataset only. Grammar content (47 topics of lesson prose, 14,100 quiz
questions) is explicitly out of scope here and will get its own separate
spec/plan later, reusing the dictionary and scan approach built in this
phase.

## Actual scope (from a full-corpus audit, not a guess)

A scripted audit of all 5,000 words (headword + definition + example +
synonyms + antonyms) against a curated US→UK term dictionary found:

- **125 of 5,000 words** need at least one field override.
- Of those, **~10 words have a headword spelling change** (e.g. flavour,
  sceptical, kerb, endeavour, instalment, catalyse) — the rest are
  definition/example/synonym/antonym text containing a US-only spelling.
- **~20-30 hits involve ambiguous pairs** (story/storey, check/cheque,
  program/programme) where the US and UK forms only differ in a specific
  sense — e.g. "story" (narrative) is spelled identically in both
  dialects, only "storey" (building floor) differs; "check" (verify)
  never becomes "cheque" (payment instrument). These require a per-
  occurrence human read, not a blind dictionary substitution — the audit
  script's naive first pass already produced wrong swaps here ("the
  storey" for a narrative, "double-cheques her reports") which will be
  hand-corrected before anything ships.

This confirms the change is a small, well-bounded content pass, not a
5,000-word retranslation.

## Language list change

`#langMenu`'s single "English" row is replaced by two rows, inserted
before "Tiếng Việt":

| Code | Label | Status |
|---|---|---|
| en | English (US) | active (default) |
| en-gb | English (UK) | active |
| vi | Tiếng Việt | active |
| zh / ja / ko / ar / fr / nl / de / es / pt / ru | (unchanged) | soon |

No other UI changes — same dropdown, same open/close/active-checkmark
behavior as today.

## State & persistence

- `state.lang` gains a third valid value, `'en-gb'`, alongside existing
  `'en'` / `'vi'`. Default stays `'en'`.
- Persisted the same way as today: `localStorage.ivm_lang`. No backend/API
  changes, no account-level sync (matches existing `en`/`vi` behavior).

## Content architecture

Mirrors the existing Vietnamese i18n architecture (`WORD_DEF_VI`,
`catName()`, etc.) rather than inventing a new pattern:

**Data:** a single override dict, `WORD_GB`, keyed by `word.pk`. Each
entry holds *only* the fields that actually differ from US, e.g.:

```js
const WORD_GB = {
  9282: { word: "flavour", example: "...flavour..." },
  5102: { example: "...double-checks her reports." }, // ambiguous case, reviewed: stays "check"
  ...
};
```

Generated by:
1. Scripted scan (curated US→UK term dictionary, whole-word match) over
   all 5,000 words' word/definition/example/synonyms/antonyms fields —
   already run once for this design's scoping numbers above.
2. Manual review of every ambiguous-pair hit (story/check/program family)
   — decide per-occurrence whether that instance's sense actually differs
   in UK English, correct or discard.
3. Spot-check the non-ambiguous auto-generated swaps for anything the
   dictionary missed or over-matched.
4. Splice the final, reviewed dict into `vocablarry.html` as static data,
   same technique used for `WORD_DEF_VI` (write intermediate JSON, verify,
   generate the JS object, splice via a unique anchor).

**Accessor:** one new helper, `ukWord(word)`:

```js
function ukWord(word){
  if (state.lang !== "en-gb") return word;
  const ov = WORD_GB[word.pk];
  return ov ? { ...word, ...ov } : word;
}
```

Every render path and quiz-generation/grading path that currently reads
`word.word` / `.def` / `.example` / `.synonyms` / `.antonyms` directly gets
routed through `ukWord(word)` first — a mechanical grep-and-fix sweep
across the codebase, the same discipline used for the `wordDef()`/
`catName()` rollout. This includes (non-exhaustive, confirmed during
implementation via full grep sweep):

- Word grid cards, word detail modal, Word/Examples page
- Category browse search/filter matching (matches the *displayed*
  spelling, consistent with how Vietnamese search already matches
  displayed Vietnamese text rather than the underlying English)
- All vocab quiz submodes (Quiz/Gap/Challenge) — both the displayed
  prompt/options and the accepted-answer comparison for typed input must
  use the UK-mode spelling when `state.lang === 'en-gb'`, so a learner
  typing "flavour" is marked correct in UK mode (and "flavor" in US mode)

**Chrome text:** no new `UI_STRINGS['en-gb']` dictionary. A grep sweep of
existing UI chrome found zero US-only spellings in nav/button/label text,
so `en-gb` inherits from `UI_STRINGS.en` via the existing fallback
(`UI_STRINGS[state.lang] || UI_STRINGS.en`). If a future chrome string
does need a UK-specific form, add it to `UI_STRINGS.en-gb` at that time —
not needed for this phase.

## Testing

- `pytest` suite: unaffected, no backend/model changes — `WORD_GB` is
  static client-side data exactly like `WORD_DEF_VI`.
- Playwright click-through (required before shipping):
  1. Switch to English (UK), confirm word cards/detail modal/definitions/
     examples/synonyms/antonyms show the swapped spelling for a sample of
     the ~10 headword-changed words and a sample of definition/example-
     only changes.
  2. Confirm the ambiguous-pair words (story/check/program family) render
     correctly in both modes — i.e. confirm the hand-reviewed decisions
     didn't introduce a wrong swap.
  3. Run a vocab quiz (at least one typed-answer submode) in UK mode on a
     headword-changed word and confirm the UK spelling is accepted as
     correct.
  4. Confirm switching back to English (US) restores the original
     spelling everywhere, and that Vietnamese mode is unaffected.

## Phase boundary

**Phase 1 (this spec):** Vocabulary word dataset (5,000 words) — language
list change, `state.lang` value, `WORD_GB` dict + `ukWord()` accessor,
full render/quiz call-site sweep, testing above.

**Phase 2 (separate spec/plan, later):** apply the same dictionary and
scan approach to Grammar content — 47 topics' lesson prose (intro/rule/
tip blocks) and 14,100 quiz questions (prompts, options, answers, "why"
explanations). Reuses Phase 1's US→UK dictionary; the ambiguous-pair
review process repeats at that corpus's own scale.

## Out of scope for Phase 1 (YAGNI)

- Grammar content (see Phase 2 above).
- Any word that only exists in one dialect's vocabulary (e.g. "trainers"/
  "eraser"/"flat"-style whole-word substitutions) — the current 5,000-word
  academic/IELTS dataset doesn't contain everyday British-only or
  American-only object nouns; this phase covers genuine spelling variants
  of words already in the dataset, not adding new vocabulary items.
- Any backend/API change, account-level sync of the US/UK choice (matches
  existing `en`/`vi` local-only behavior).
- A live/regex-based runtime transform — rejected in favor of a reviewed
  static dict, since blind substitution demonstrably produces wrong output
  on the ambiguous pairs (see scope section above).
